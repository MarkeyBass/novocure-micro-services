# 3.5 — BookCreated: Publish from BookStoreApi (MassTransit), Consume in TodoApi (MassTransit)

This slice introduces **MassTransit** as the message bus abstraction for the first time.
When a new book is created via `POST /api/books`, BookStoreApi publishes a `BookCreated` event.
TodoApi consumes it and inserts a SQL todo row: `"Read book: {BookName}"`.

The existing raw `RabbitMQ.Client` flows (slices 3.1–3.4) are intentionally left untouched —
both patterns coexist in this repo so they serve as side-by-side reference implementations.

---

## Why MassTransit?

| Concern | Raw `RabbitMQ.Client` (slices 3.1–3.4) | MassTransit (this slice) |
|---|---|---|
| Connection management | Manual `ConnectionFactory`, `CreateConnection`, `CreateModel` | Managed internally |
| Exchange + queue declare | Explicit `ExchangeDeclare`, `QueueDeclare`, `QueueBind` in consumer | Auto-declared from consumer registration |
| Serialization | Manual `JsonSerializer.Serialize` → bytes → publish | Handled internally |
| Ack / Nack | Explicit `channel.BasicAck` / `channel.BasicNack` | Automatic (exception → nack, success → ack) |
| Consumer lifecycle | `BackgroundService` + `RunConsumerLoop` + `stoppingToken.WaitHandle.WaitOne()` | Implement `IConsumer<T>` — one method |
| DI scope per message | Manual `IServiceScopeFactory.CreateScope()` | Built-in (MassTransit creates a scope per message) |
| Retry / DLQ | Not wired yet | Configurable via `UseMessageRetry` / `UseDelayedRedelivery` |

**Net reduction: ~100 lines of infrastructure boilerplate per producer/consumer pair.**

  - `IsComplete = false`
  - `BookId = context.Message.BookId`

- `await _context.SaveChangesAsync()`

### Step 8 — Containerize BookStoreApi

- Add `BookStoreApi/Dockerfile` (same multi-stage pattern as TodoApi/HousingApi)

- Add `MassTransitRabbitMQ__*` env vars to existing `todo-api` service entry

### Step 9 — RabbitMQ topology created automatically by MassTransit

MassTransit creates **two exchanges** per message type:

```
book.created         (fanout, durable)  ← producer publishes here
  └─→ todo-api.book-created  (fanout, durable)  ← endpoint exchange
        └─→ todo-api.book-created  (queue, durable)  ← messages land here
```

Why two exchanges? MassTransit's routing model uses one exchange per message type
and one exchange per receive endpoint, then chains them. This allows multiple consumers
to subscribe to the same event without queue coupling. See the doc section below for
how to verify this in the management UI.

### Consumer

**Raw (`HousingApplicationCreatedConsumer.cs` — ~80 lines):**
```csharp
public class HousingApplicationCreatedConsumer : BackgroundService
{
    protected override Task ExecuteAsync(CancellationToken stoppingToken) =>
        Task.Run(() => RunConsumerLoop(stoppingToken), stoppingToken);

    private void RunConsumerLoop(CancellationToken stoppingToken)
    {
        var factory = new ConnectionFactory { ... };
        using var connection = factory.CreateConnection();
        using var channel = connection.CreateModel();
        channel.ExchangeDeclare(...);
        channel.QueueDeclare(...);
        channel.QueueBind(...);
        channel.BasicQos(prefetchSize: 0, prefetchCount: 1, global: false);
        var consumer = new EventingBasicConsumer(channel);
        consumer.Received += (_, ea) =>
        {

            channel.BasicAck(ea.DeliveryTag, multiple: false);
```

## RabbitMQ topology

```
[BookStoreApi] --publish--> book.created (fanout, durable)
                                  │
                             [MassTransit binding]
                                  │
                            todo-api.book-created (fanout, durable)
                                  │
                            todo-api.book-created (queue, durable)
                                  │
                            [TodoApi BookCreatedConsumer]
```

Two exchanges are always present with MassTransit:
1. **`book.created`** — message-type exchange; the publisher targets this
2. **`todo-api.book-created`** — endpoint exchange; bound to `book.created`; the queue is bound to this

This indirection means a second consumer service (e.g. a future `NotificationApi`) can bind
its own endpoint exchange to `book.created` without touching the existing queue topology.

## Caveats / what comes next

| Caveat | Impact | Next step |
|---|---|---|
| No retry policy configured | Exception in `BookCreatedConsumer` causes MassTransit to nack and move message to `_error` queue after default retries | Add `e.UseMessageRetry(r => r.Intervals(500, 1000, 5000))` inside `ReceiveEndpoint` config |
