# 3.2 — RabbitMQ Consumer (TodoApi)

This document covers the second RabbitMQ slice: wiring TodoApi as a consumer of the `HousingApplicationCreated` event published by HousingApi.

---

## Agenda

- Add the TodoApi consumer for `HousingApplicationCreated`
- Declare a durable queue in TodoApi
- Bind it to `housing.application.created`
- When the event arrives, create a SQL todo row like:
  - `Review housing application for Mark Tester`

### [`TodoApi/Messages/HousingApplicationCreated.cs`](../TodoApi/Messages/HousingApplicationCreated.cs) *(new file)*

```csharp
namespace TodoApi.Messages;

// Mirror of HousingApi.Messages.HousingApplicationCreated.
// Both sides define the contract independently — no shared library.
// Field names and types must stay in sync with the producer.
public record HousingApplicationCreated(
    string ApplicationId,
    string HousingId,
    string FirstName,
    string LastName,
    string Email,
    DateTime CreatedAt
);
```

**Why duplicate instead of sharing?**
A shared contracts library would require TodoApi and HousingApi to reference the same project or NuGet package. That creates a build-time coupling between two independent services — if the shared package changes, both must update and redeploy together. The microservices approach is to duplicate the contract in each service and treat the wire format (JSON field names) as the real contract. This is "consumer-driven contract" style: TodoApi only cares about the fields it uses; if HousingApi adds a field, TodoApi keeps working without changes.

### [`TodoApi/Consumers/HousingApplicationCreatedConsumer.cs`](../TodoApi/Consumers/HousingApplicationCreatedConsumer.cs) *(new file)*

This is the core of the slice. Broken down by concern:

#### Class declaration

```csharp
public class HousingApplicationCreatedConsumer : BackgroundService
```

`BackgroundService` is ASP.NET Core's abstract base class for long-running hosted services. It implements `IHostedService` and exposes one abstract method: `ExecuteAsync(CancellationToken)`. The runtime calls `StartAsync` on app startup and `StopAsync` on shutdown, which triggers the cancellation token.

#### Why `IServiceScopeFactory` instead of injecting `TodoContext` directly

```csharp
private readonly IServiceScopeFactory _scopeFactory;
```

`AddDbContext<TodoContext>` registers `TodoContext` as **scoped** (one instance per HTTP request). `BackgroundService` is effectively a **singleton** — it lives for the app's lifetime. Injecting a scoped service into a singleton causes a "captive dependency" bug: the first `TodoContext` instance is captured and reused forever, which means EF Core's change tracker accumulates state across messages, leading to incorrect saves or crashes.

The fix is `IServiceScopeFactory`: create a new scope per message, get a fresh `TodoContext` from that scope, and dispose both when done:

```csharp
using var scope = _scopeFactory.CreateScope();
var db = scope.ServiceProvider.GetRequiredService<TodoContext>();
db.TodoItems.Add(todo);
db.SaveChanges();
// scope is disposed here — TodoContext is disposed with it
```

#### `Task.Run` wrapping `ExecuteAsync`

```csharp
protected override Task ExecuteAsync(CancellationToken stoppingToken)
{
    return Task.Run(() => RunConsumerLoop(stoppingToken), stoppingToken);
}
```

`ExecuteAsync` must return promptly during startup, otherwise the host waits for it before marking the app as started. Since `RunConsumerLoop` blocks (it holds an open TCP connection and waits for messages), wrapping it in `Task.Run` moves it to the thread pool and lets startup complete. Without `Task.Run`, the app would hang at startup.

#### Exchange + queue + binding

```csharp
channel.ExchangeDeclare(exchange: ExchangeName, type: ExchangeType.Fanout, durable: true);

channel.QueueDeclare(queue: QueueName, durable: true, exclusive: false, autoDelete: false, arguments: null);

channel.QueueBind(queue: QueueName, exchange: ExchangeName, routingKey: "");
```

- **`ExchangeDeclare`** — idempotent, must match the producer's declaration exactly (same `type` and `durable`). If the exchange already exists with different parameters, RabbitMQ throws. Since HousingApi declares it first, this call is a no-op.
- **`QueueDeclare`** — creates the queue if it doesn't exist. `durable: true` means the queue survives a RabbitMQ restart. `autoDelete: false` means the queue stays alive even when TodoApi is not running — messages accumulate and are consumed when the service comes back.
- **`QueueBind`** — attaches the queue to the exchange. After this call, every message published to `housing.application.created` is routed to `todo-api.housing-application-created`.

**Queue name:** `todo-api.housing-application-created` — follows the `{consumer}.{exchange}` convention, making it immediately clear from the RabbitMQ UI which service owns the queue and what event it handles.

#### Manual ack/nack

```csharp
channel.BasicConsume(queue: QueueName, autoAck: false, consumer: consumer);
```

`autoAck: false` means RabbitMQ keeps the message in "unacknowledged" state until the consumer explicitly calls `BasicAck` or `BasicNack`. This is the safe pattern:

- **Success**: `BasicAck` — message is removed from the queue.
- **Failure**: `BasicNack(requeue: true)` — message goes back to the front of the queue for retry.

With `autoAck: true`, the message is removed from the queue the moment it is delivered, even if processing fails.

#### `BasicQos(prefetchCount: 1)`

```csharp
channel.BasicQos(prefetchSize: 0, prefetchCount: 1, global: false);
```

Tells RabbitMQ to deliver at most 1 unacknowledged message at a time to this consumer. Without this, RabbitMQ could push the entire queue backlog to the consumer at once. With one DB write per message, processing them sequentially is the right default.

#### Blocking until shutdown

```csharp
stoppingToken.WaitHandle.WaitOne();
```

After the consumer is registered, the thread blocks here. When the app shuts down, ASP.NET Core signals the cancellation token, `WaitOne()` returns, the `using` blocks dispose the channel and connection, and the method returns.

### [`TodoApi/Program.cs`](../TodoApi/Program.cs)

**What was added:**
```csharp
using TodoApi.Consumers;

// RabbitMQ — bind settings and register the background consumer.
// The consumer starts automatically when the app starts (IHostedService contract).
builder.Services.Configure<RabbitMqSettings>(builder.Configuration.GetSection("RabbitMQ"));
builder.Services.AddHostedService<HousingApplicationCreatedConsumer>();
```

`AddHostedService` registers the consumer so ASP.NET Core calls `StartAsync` (which calls `ExecuteAsync`) when the app starts. The consumer runs for the full lifetime of the process.

### [`med-tech-local-infra/docker-compose.yml`](../med-tech-local-infra/docker-compose.yml)

**What was added to `todo-api`:**
```yaml
environment:
  ...
  RabbitMQ__HostName: rabbitmq      # ← new

depends_on:
  sqlserver:
    condition: service_healthy      # unchanged
  rabbitmq:
    condition: service_started      # ← new
```

`service_started` (not `service_healthy`) is used for RabbitMQ because no healthcheck is defined for it. `service_started` only ensures the container has been started, not that RabbitMQ is ready to accept connections. The consumer begins connecting during app startup via `AddHostedService`/`ExecuteAsync`, so there is still a small readiness race here. In practice it usually works locally, but the caveats section correctly calls out that a retry loop or RabbitMQ healthcheck would make this more robust.

## Caveats / what comes next

| Caveat | Impact | Next step |
|---|---|---|
| `requeue: true` on failure with no DLQ | A poison message (e.g. malformed JSON) is nacked and immediately requeued, causing an infinite retry loop that pegs CPU | Add a dead-letter exchange + DLQ, or add a retry counter check |
| Consumer connects at startup with no retry | If RabbitMQ is still initialising when `ExecuteAsync` fires, the connection throws and the consumer dies silently | Wrap `RunConsumerLoop` in a retry loop with backoff |
| No reverse event (`TodoCompleted`) | The roadmap calls for HousingApi to react when a review todo is completed — not yet implemented | Next slice: TodoApi publishes `TodoCompleted`; HousingApi consumes it |
| No BookStore events | `BookCreated` producer in BookStoreApi and its TodoApi consumer are not yet done | Follow the same pattern as this slice |
