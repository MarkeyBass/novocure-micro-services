# 3.1 ŌĆö RabbitMQ Config & Producer (HousingApi)

This document covers the first RabbitMQ slice: configuring HousingApi to connect to RabbitMQ and publish a `HousingApplicationCreated` event after a housing application is persisted in MongoDB.

No consumer is implemented yet. The exchange is declared, the event is published, and the next step (TodoApi consumer) can bind a queue to it.

### [`HousingApi/HousingApi.csproj`](../HousingApi/HousingApi.csproj)

**CLI command used:**
```bash
cd HousingApi
dotnet add package RabbitMQ.Client --version "6.*"
```

This resolves and adds the latest `6.x` release (landed on `6.8.1`) to the `.csproj`. You never edit the `.csproj` by hand for packages ŌĆö `dotnet add package` writes the `<PackageReference>` entry and updates `obj/project.assets.json`.

**Why `RabbitMQ.Client` and not MassTransit?**
MassTransit is a higher-level framework that abstracts exchange/queue topology, retry, dead-letter queues, and sagas. That abstraction is valuable at scale but hides how RabbitMQ actually works. For a first learning slice, raw `RabbitMQ.Client` keeps every decision visible: which exchange, which type, which routing key, which properties on the message. MassTransit is the natural upgrade path once the consumer is in place and retry/DLQ behavior is needed.

**What was added to the file:**
```xml
<PackageReference Include="RabbitMQ.Client" Version="6.*" />
```

### [`HousingApi/Messages/HousingApplicationCreated.cs`](../HousingApi/Messages/HousingApplicationCreated.cs) *(new file)*

```csharp
namespace HousingApi.Messages;

public record HousingApplicationCreated(
    string ApplicationId,
    string HousingId,
    string FirstName,
    string LastName,
    string Email,
    DateTime CreatedAt
);
```

**What it is:**
The *message contract* ŌĆö the shape of the event that travels over the wire. It lives in a dedicated `Messages/` folder to keep it separate from database models (`Models/`) and business logic (`Services/`). When we add a consumer in TodoApi, it will define an identical record in its own `Messages/` folder, making the contract explicit on both sides without any shared library.

**Why a `record` and not a `class`?**
Records are immutable by default and have value-based equality. Events should never be mutated after they are created, so a record is the right type. They also serialize cleanly to JSON.

**Field rationale:**
Every field a consumer needs to act without making an extra HTTP call back to HousingApi is included. `ApplicationId` and `HousingId` are the Mongo ObjectId strings assigned after the insert.

### [`HousingApi/Models/RabbitMqSettings.cs`](../HousingApi/Models/RabbitMqSettings.cs) *(new file)*

```csharp
public class RabbitMqSettings
{
    public string HostName { get; set; } = "localhost";
    public string UserName { get; set; } = "admin";
    public string Password { get; set; } = "admin";
}
```

**What it does:**
Holds the values that change between environments (host dev vs Docker Compose). ASP.NET Core's options pattern binds this class to the `RabbitMQ` section in `appsettings.json` at startup, then injects it anywhere via `IOptions<RabbitMqSettings>`.

The property defaults (`localhost`, `admin`, `admin`) match the host-dev environment so the developer does not need to set env vars when running `dotnet watch run`. In Docker Compose, the `RabbitMQ__HostName=rabbitmq` env var overrides just the hostname ŌĆö the double underscore (`__`) is how ASP.NET Core maps environment variable names to nested config keys (`RabbitMQ:HostName`).

### [`HousingApi/Services/RabbitMqPublisher.cs`](../HousingApi/Services/RabbitMqPublisher.cs) *(new file)*

This is the core of the slice. Full annotated version:

```csharp
// Registered as a singleton ŌĆö one connection shared for the lifetime of the app.
// IDisposable so ASP.NET Core disposes the connection cleanly on shutdown.
public class RabbitMqPublisher : IDisposable
{
    private const string ExchangeName = "housing.application.created";

    private readonly IConnection _connection;
    private readonly IModel _channel;
    private readonly ILogger<RabbitMqPublisher> _logger;

    public RabbitMqPublisher(IOptions<RabbitMqSettings> options, ILogger<RabbitMqPublisher> logger)
    {
        _logger = logger;

        var factory = new ConnectionFactory
        {
            HostName = options.Value.HostName,
            UserName = options.Value.UserName,
            Password = options.Value.Password,
        };

        _connection = factory.CreateConnection();
        _channel = _connection.CreateModel();

        // durable: true ŌĆö exchange survives a RabbitMQ restart
        _channel.ExchangeDeclare(exchange: ExchangeName, type: ExchangeType.Fanout, durable: true);

        _logger.LogInformation("RabbitMQ connected to {HostName}, exchange '{Exchange}' ready", ...);
    }

    public void Publish(HousingApplicationCreated message)
    {
        var body = Encoding.UTF8.GetBytes(JsonSerializer.Serialize(message));

        var props = _channel.CreateBasicProperties();
        props.Persistent = true;           // message survives broker restart
        props.ContentType = "application/json";

        // Fanout exchange ignores the routing key ŌĆö pass empty string by convention.
        _channel.BasicPublish(exchange: ExchangeName, routingKey: "", basicProperties: props, body: body);

        _logger.LogInformation("Published HousingApplicationCreated: ApplicationId={...}, HousingId={...}", ...);
    }

    public void Dispose() { _channel?.Dispose(); _connection?.Dispose(); }
}
```

**Key decisions explained:**

**`IConnection` + `IModel` (channel) as fields:**
In RabbitMQ.Client v6, `IConnection` is a TCP connection to the broker. `IModel` is a "channel" ŌĆö a lightweight multiplexed session inside the connection. One connection with one long-lived channel is correct for a publisher that fires infrequently. (High-throughput producers sometimes pool channels, but that is unnecessary here.)

**`ExchangeDeclare` in the constructor:**
The producer declares the exchange with `durable: true`. "Declaring" in RabbitMQ is idempotent ŌĆö if the exchange already exists with the same parameters, the call succeeds silently. If it doesn't exist, RabbitMQ creates it. This means starting either the producer or the consumer first is safe; whoever starts first creates the exchange.

**`ExchangeType.Fanout`:**
A fanout exchange delivers every message to every queue bound to it, regardless of routing key. This is the correct type for a domain event where multiple consumers may eventually be interested (TodoApi, BookStoreApi, analytics, etc.) ŌĆö none of the consumers need to be known at publish time.

**`props.Persistent = true`:**
Tells RabbitMQ to write the message to disk before acknowledging. Without this, messages in a durable queue are lost if the broker restarts before a consumer reads them. Combined with `durable: true` on the exchange (and on the queue, which the consumer will declare), this gives basic durability.

**Why `_publisher` connects in the constructor (lazy singleton):**
The publisher is registered as a singleton and constructed the first time the DI container resolves it ŌĆö which is on the first HTTP request to the controller. It does not connect at app startup. This is acceptable for a first slice but means: if RabbitMQ is unreachable, the *first* POST to `/api/applications` fails with 500. A production-grade fix is an `IHostedService` that connects eagerly with retry, or a reconnect strategy on the channel.

### [`HousingApi/Program.cs`](../HousingApi/Program.cs)

**What was added:**
```csharp
builder.Services.Configure<RabbitMqSettings>(builder.Configuration.GetSection("RabbitMQ"));
builder.Services.AddSingleton<RabbitMqPublisher>();
```

`Configure<T>` binds the `RabbitMQ` config section to `RabbitMqSettings` and registers it so it can be injected as `IOptions<RabbitMqSettings>`.

`AddSingleton<RabbitMqPublisher>` registers the publisher as a singleton ŌĆö one instance, one connection, shared for the lifetime of the app. A transient or scoped lifetime would create a new TCP connection on every request, which would be wasteful and incorrect.

### [`HousingApi/Controllers/ApplicationsController.cs`](../HousingApi/Controllers/ApplicationsController.cs)

**What was added to the constructor:**

```csharp
private readonly RabbitMqPublisher _publisher;

public ApplicationsController(
    HousingApplicationsService applicationsService,
    HousingLocationsService locationsService,
    RabbitMqPublisher publisher          // ŌåÉ new
)
{
    ...
    _publisher = publisher;
}
```

**What was added to the POST handler:**
```csharp
// Persist to Mongo first. If this throws, the event is never published.
await _applicationsService.CreateAsync(application);

// Publish only after a successful insert ŌĆö application.Id is now set by Mongo.
_publisher.Publish(new HousingApplicationCreated(
    ApplicationId: application.Id!,
    HousingId:     application.HousingId,
    FirstName:     application.FirstName,
    LastName:      application.LastName,
    Email:         application.Email,
    CreatedAt:     application.CreatedAt
));
```

**Ordering is deliberate:**
`CreateAsync` throws if the Mongo insert fails ŌĆö in that case execution never reaches `Publish`. The event is only fired after the document exists in the database. The reverse order would risk publishing an event for a record that was never actually saved.

**What `application.Id!` means:**
After `InsertOneAsync` returns, MongoDB's C# driver has populated `application.Id` with the newly assigned ObjectId string. The `!` is a null-forgiveness operator ŌĆö it tells the compiler "I know this is not null here even though the type is `string?`". This is safe because the MongoDB driver always sets the `[BsonId]` field after a successful insert.

### [`med-tech-local-infra/docker-compose.yml`](../med-tech-local-infra/docker-compose.yml)

**What was added to the `housing-api` service:**
```yaml
environment:
  ...
  RabbitMQ__HostName: rabbitmq    # ŌåÉ new
depends_on:
  mongo:
    condition: service_started
  rabbitmq:
    condition: service_healthy    # ŌåÉ new (RabbitMQ healthcheck added in 5.2)
```

`RabbitMQ__HostName: rabbitmq` overrides `appsettings.json` at runtime. `rabbitmq` is the Compose service name, which Docker's internal DNS resolves to the container's IP on `app-network`.

`condition: service_healthy` tells Compose to wait until the RabbitMQ healthcheck (`rabbitmq-diagnostics ping`) passes before starting `housing-api`. This prevents the consumer from crashing on startup when RabbitMQ is still initialising. A `ConnectWithRetry` loop in the consumer provides an additional safety net (added in slice 5.2).

## Caveats / what comes next

| Caveat | Impact | Next step |
|---|---|---|
| No queue bound to the exchange yet | Messages published before a consumer binds a queue are silently dropped by RabbitMQ | TodoApi consumer declares queue + binds on startup |
| Publisher connects lazily | First POST fails with 500 if RabbitMQ is unreachable | Add `IHostedService` with eager connect + retry |
| No outbox | Mongo insert can succeed but publish can fail, leaving no record | Add transactional outbox pattern when reliability matters |
| Credentials in `appsettings.json` | Fine for local dev, not for production | Move to secrets / env vars before any real deployment |
