## Detailed breakdown

---

### What had to be done before running the migration command

The migration command does **not** invent schema changes by itself. EF Core scaffolds a migration by comparing:

1. the **current model in code**
2. the **previous model snapshot**

So before running `dotnet ef migrations add ...`, the model first had to change in code.

In this slice that meant adding these properties to `TodoItem`:

```csharp
public string? HousingApplicationId { get; set; }
public DateTime? CreatedAt { get; set; }
public DateTime? UpdatedAt { get; set; }
public DateTime? CompletedAt { get; set; }
```

If only the migration command had been run without changing `TodoItem`, EF Core would likely have generated an empty migration or refused to scaffold meaningful changes.

The DTO, controller, and consumer changes were also manual. Those were not required just to create the migration file, but they were required to make the new columns useful in real application behavior.

---

### CLI commands used

**Build first:**
```bash
dotnet build "TodoApi/TodoApi.csproj"
```

**Create the migration:**
```bash
dotnet ef migrations add AddTodoMetadataFields \
  --project "TodoApi/TodoApi.csproj" \
  --startup-project "TodoApi/TodoApi.csproj"
```

Why both flags point to `TodoApi`:

- `--project` tells EF where the `DbContext` and migrations live
- `--startup-project` tells EF which app to run for design-time configuration

That matters because EF Core needs to create `TodoContext` at design time using the same app configuration it would use at runtime.

---

### What the migration command actually did internally

When `dotnet ef migrations add AddTodoMetadataFields` ran, EF Core did roughly this:

1. Restored and built the project so it could load the compiled assembly.
2. Found `TodoContext`.
3. Built the **current EF model** from your code (`TodoItem`, `TodoContext`, provider config, etc.).
4. Read the previous snapshot from `TodoContextModelSnapshot.cs`.
5. Compared old snapshot vs current model.
6. Detected 4 new nullable columns on `TodoItems`.
7. Generated a new migration file with the SQL-intent operations.
8. Generated the matching `.Designer.cs` file.
9. Rewrote `TodoContextModelSnapshot.cs` so the snapshot now represents the latest model state.

Important: this command **did not** directly alter the database schema. It only generated C# migration artifacts.

The schema changes are applied later by either:

- `dotnet ef database update`, or
- app startup, because `TodoApi/Program.cs` already calls `db.Database.Migrate();`

### [`TodoApi/Models/TodoItem.cs`](../TodoApi/Models/TodoItem.cs)

**Manual change**

This is the EF Core entity model. Changing this file is what told EF Core that the SQL schema should change.

**What was added:**
```csharp
public string? HousingApplicationId { get; set; }
public DateTime? CreatedAt { get; set; }
public DateTime? UpdatedAt { get; set; }
public DateTime? CompletedAt { get; set; }
```

How EF interpreted those:

- `string?` -> nullable SQL string column (`nvarchar(max)`)
- `DateTime?` -> nullable SQL datetime column (`datetime2`)

Because all four properties are nullable, the generated migration used `nullable: true` for each new column.

---

### [`TodoApi/Models/TodoItemDTO.cs`](../TodoApi/Models/TodoItemDTO.cs)

**Manual change**

This file is **not** used by EF Core to generate the migration. It exists for the HTTP API shape.

It was still important to update it so `GET /api/TodoItems` returns the new fields and `POST`/`PUT` can receive them when needed.

Without this DTO change, the database could have the new columns while the API still hides them.

---

### [`TodoApi/Controllers/TodoItemsController.cs`](../TodoApi/Controllers/TodoItemsController.cs)

**Manual change**

This file is also **not** part of migration generation. It was changed so the new fields are actually maintained by the application.

Key additions:

- `POST` now sets `CreatedAt` and `UpdatedAt`
- `POST` sets `CompletedAt` immediately if the todo is created already complete
- `PUT` refreshes `UpdatedAt`
- `PUT` sets or clears `CompletedAt` based on `IsComplete`
- `ItemToDTO(...)` now includes the new fields in responses
- `HousingApplicationId` is normalized so blank strings become `null`

This is an important learning point:

- **EF model change** controls the schema
- **controller/service change** controls application behavior

You usually need both in a real feature.

---

### [`TodoApi/Consumers/HousingApplicationCreatedConsumer.cs`](../TodoApi/Consumers/HousingApplicationCreatedConsumer.cs)

**Manual change**

This was not required for the migration itself, but it was needed for consistency.

Before the change, the RabbitMQ consumer created todos without populating `HousingApplicationId`, `CreatedAt`, or `UpdatedAt`. That would have left the new columns mostly unused for event-created rows.

Now it sets:

```csharp
HousingApplicationId = message.ApplicationId,
CreatedAt = now,
UpdatedAt = now,
```

So both HTTP-created todos and RabbitMQ-created todos follow the same metadata pattern.

---

### [`TodoApi/Migrations/20260418080717_AddTodoMetadataFields.cs`](../TodoApi/Migrations/20260418080717_AddTodoMetadataFields.cs)

**CLI-generated**

This is the main migration file generated by EF Core.

It contains two methods:

- `Up(...)` — how to move the schema forward
- `Down(...)` — how to roll the schema back

In `Up(...)`, EF added 4 columns:

```csharp
migrationBuilder.AddColumn<DateTime>(name: "CompletedAt", table: "TodoItems", type: "datetime2", nullable: true);
migrationBuilder.AddColumn<DateTime>(name: "CreatedAt", table: "TodoItems", type: "datetime2", nullable: true);
migrationBuilder.AddColumn<string>(name: "HousingApplicationId", table: "TodoItems", type: "nvarchar(max)", nullable: true);
migrationBuilder.AddColumn<DateTime>(name: "UpdatedAt", table: "TodoItems", type: "datetime2", nullable: true);
```

In `Down(...)`, EF generated the reverse operations with `DropColumn(...)`.

This file is the one you would usually review most closely because it is the clearest representation of the schema delta.

---

### [`TodoApi/Migrations/20260418080717_AddTodoMetadataFields.Designer.cs`](../TodoApi/Migrations/20260418080717_AddTodoMetadataFields.Designer.cs)

**CLI-generated**

This file is auto-generated metadata for the migration. It captures the **target EF model** at the point this migration was created.

You normally do **not** edit this by hand.

Why it exists:

- it helps EF understand the model associated with this migration
- it supports future diffing and tooling behavior

Think of it as EF's bookkeeping for the migration rather than the human-authored schema change itself.

---

### [`TodoApi/Migrations/TodoContextModelSnapshot.cs`](../TodoApi/Migrations/TodoContextModelSnapshot.cs)

**CLI-updated**

This is the most important EF bookkeeping file in the migrations folder.

It represents the **latest known model state** after all migrations.

Before this slice, the snapshot did not include:

- `HousingApplicationId`
- `CreatedAt`
- `UpdatedAt`
- `CompletedAt`

After the command ran, EF rewrote the snapshot so those properties are now part of the current model baseline.

Why EF updates it:

- next time you run `dotnet ef migrations add ...`, EF compares your code model against this snapshot
- the snapshot is how EF knows what changed since the last migration

So the "2 new files + 1 updated file" pattern is normal:

- new migration `.cs`
- new migration `.Designer.cs`
- updated `TodoContextModelSnapshot.cs`

## Manual vs CLI-generated summary

| File or action | Manual or CLI? | Why |
|---|---|---|
| Add properties to `TodoItem` | Manual | Defines the new EF model |
| Add properties to `TodoItemDTO` | Manual | Exposes fields via API contract |
| Update `TodoItemsController` | Manual | Makes app read/write the fields |
| Update `HousingApplicationCreatedConsumer` | Manual | Makes event-created rows populate the fields |
| `dotnet build` | CLI | Validates project builds before scaffolding |
| `dotnet ef migrations add AddTodoMetadataFields ...` | CLI | Compares model vs snapshot and scaffolds migration artifacts |
| Create migration `.cs` | CLI | Generated by EF Core |
| Create migration `.Designer.cs` | CLI | Generated by EF Core |
| Update `TodoContextModelSnapshot.cs` | CLI | Generated by EF Core |
| Apply schema to actual database | Runtime / CLI | Happens later via `Database.Migrate()` or `dotnet ef database update` |

## Caveats / learning notes

| Topic | Why it matters | Takeaway |
|---|---|---|
| Model change vs migration file | Editing only the migration file is not the normal workflow | First change the model, then scaffold the migration |
| DTO/controller changes do not affect EF schema diffing | EF only looks at the model, not your API DTOs | Keep persistence concerns and API concerns distinct |
| Migration creation does not update the DB | It only creates C# artifacts | The DB changes later via `database update` or `Database.Migrate()` |
| Snapshot is critical | EF uses it as the baseline for future diffs | Do not hand-edit the snapshot unless you know exactly why |
| Nullable properties produced nullable columns | That is why no default value or data backfill was needed | `string?` / `DateTime?` maps naturally to optional SQL columns |
