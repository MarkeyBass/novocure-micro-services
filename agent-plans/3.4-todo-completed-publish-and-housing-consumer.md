# 3.4 — TodoCompleted: Publish from TodoApi, Consume in HousingApi

This document covers the third RabbitMQ slice: TodoApi publishes a `TodoCompleted` event when a linked todo transitions to complete, and HousingApi consumes it to increment a review counter on the housing application document.

---

## Agenda

- Publish `TodoCompleted` from `TodoApi`
- Only publish when a todo transitions from `IsComplete = false` → `IsComplete = true`
- Include `HousingApplicationId` in the event; publish only when it is valid/present
- Add a `HousingApi` consumer for `TodoCompleted`
- Declare a durable queue in `HousingApi` and bind it to the `TodoCompleted` exchange
- When the event arrives and `housingApplicationId` is valid, increment `CompletedReviewCount` on the housing application document

## Detailed breakdown

---

### [`TodoApi/Messages/TodoCompleted.cs`](../TodoApi/Messages/TodoCompleted.cs) *(new)*

```csharp
public record TodoCompleted(
    long TodoId,
    string HousingApplicationId,
    DateTime CompletedAt,
    string? Name
);
```

**Field rationale:**
- `TodoId` — identifies which todo triggered the event; useful for deduplication later
- `HousingApplicationId` — the Mongo ObjectId of the application being reviewed; HousingApi uses this to locate the document to update
- `CompletedAt` — the exact timestamp written to SQL, used as `LastReviewedAt` on the housing document
- `Name` — optional, useful for logging on the consumer side without another DB call

`HousingApplicationId` is `string` (not `string?`) in the event because the publisher only fires when it is non-null/non-whitespace — by the time the message is constructed, it is guaranteed present.

### [`HousingApi/Messages/TodoCompleted.cs`](../HousingApi/Messages/TodoCompleted.cs) *(new)*

Identical record, independently declared. Same reasoning as the `HousingApplicationCreated` contract mirror: no shared library, wire format (JSON field names) is the contract. If `TodoApi` adds a field, `HousingApi` keeps deserializing with `PropertyNameCaseInsensitive = true` and simply ignores the unknown field.

### [`TodoApi/Services/RabbitMqPublisher.cs`](../TodoApi/Services/RabbitMqPublisher.cs) *(new)*

```csharp
public class RabbitMqPublisher : IDisposable
{
    private const string ExchangeName = "todo.completed";
    // ...
    public void Publish(TodoCompleted message) { ... }
}
```

Same pattern as `HousingApi/Services/RabbitMqPublisher.cs`:
- Singleton, one TCP connection + one channel for the app lifetime
- Declares the fanout exchange `todo.completed` (durable) in the constructor
- Publishes JSON with `Persistent = true` — message survives broker restart
- `IDisposable` so ASP.NET Core disposes the connection on shutdown

**Why a separate publisher class rather than extending HousingApi's publisher?**
Each publisher owns exactly one exchange. Mixing two exchanges into one class would require two channels and two exchange declarations, making the class harder to reason about. One publisher per exchange keeps concerns separated.

### [`TodoApi/Controllers/TodoItemsController.cs`](../TodoApi/Controllers/TodoItemsController.cs)

**Constructor change:**
```csharp
public TodoItemsController(
    TodoContext context,
    RabbitMqPublisher publisher,
    ILogger<TodoItemsController> logger)
```

**`PutTodoItem` change — transition detection:**

```csharp
// Capture the pre-update state before any mutation.
var wasComplete = todoItem.IsComplete;

// ... apply all field updates ...
await _context.SaveChangesAsync();

// Only publish on false → true transition.
if (!wasComplete && todoItem.IsComplete)
{
    var housingAppId = todoItem.HousingApplicationId;
    if (!string.IsNullOrWhiteSpace(housingAppId))
    {
        _publisher.Publish(new TodoCompleted(
            TodoId: todoItem.Id,
            HousingApplicationId: housingAppId,
            CompletedAt: todoItem.CompletedAt ?? now,
            Name: todoItem.Name
        ));
    }
    else
    {
        _logger.LogInformation(
            "Todo {TodoId} completed but has no HousingApplicationId — skipping publish", ...);
    }
}
else
{
    _logger.LogInformation(
        "Todo {TodoId} updated (wasComplete={WasComplete}, isComplete={IsComplete}) — no completion transition, skipping publish", ...);
}
```

**Why capture `wasComplete` before mutating `todoItem.IsComplete`?**
`FindAsync` returns the entity tracked by EF Core. When we do `todoItem.IsComplete = todoDTO.IsComplete`, the old value is gone. We must read `todoItem.IsComplete` into `wasComplete` before that line, or we lose the ability to detect the transition.

**When publish does NOT fire:**
| Scenario | `wasComplete` | `todoItem.IsComplete` | Publishes? |
|---|---|---|---|
| `false → false` | false | false | No |
| `false → true` | false | true | Yes (if HousingApplicationId set) |
| `true → true` | true | true | No |
| `true → false` | true | false | No |
| Any, no HousingApplicationId | — | true | No (logged) |

**Publish happens after `SaveChangesAsync`** — if the DB write fails, the event is never published. The reverse (publish before save) would risk notifying consumers about a record that was never committed.

### [`HousingApi/Models/HousingApplicationEntity.cs`](../HousingApi/Models/HousingApplicationEntity.cs)

**What was added:**
```csharp
public int CompletedReviewCount { get; set; }
public DateTime? LastReviewedAt { get; set; }
```

**Why these two fields and not something more complex?**
The requirement is "update housing-side state/counter." The minimal useful counter is a single `int`. `LastReviewedAt` adds the timestamp of the most recent review with no extra cost. Together they answer "has this application been reviewed?" and "when was it last reviewed?" without any schema redesign.

**No MongoDB migration needed.** MongoDB is schemaless — existing documents without `CompletedReviewCount` deserialize to `0` (C# `int` default) and `LastReviewedAt` to `null`. The first `IncrementCompletedReviewAsync` call writes these fields into the document using `$inc` / `$set`, not a schema migration.

### [`HousingApi/Services/HousingApplicationsService.cs`](../HousingApi/Services/HousingApplicationsService.cs)

**What was added:**
```csharp
public async Task IncrementCompletedReviewAsync(string id, DateTime reviewedAt)
{
    var update = Builders<HousingApplicationEntity>.Update
        .Inc(x => x.CompletedReviewCount, 1)
        .Set(x => x.LastReviewedAt, reviewedAt);

    await _applicationsCollection.UpdateOneAsync(x => x.Id == id, update);
}
```

**Why `$inc` + `$set` instead of FindAsync → modify → ReplaceOneAsync?**
`$inc` is an atomic MongoDB server-side operation. FindAsync → modify → Replace is a read-modify-write cycle that has a race condition: two events arriving simultaneously could both read `count = 0`, both write `count = 1`, and lose one increment. `$inc` avoids this by delegating the increment to the server.

### [`HousingApi/Consumers/TodoCompletedConsumer.cs`](../HousingApi/Consumers/TodoCompletedConsumer.cs) *(new)*

Same `BackgroundService` pattern as `TodoApi`'s consumer:

```csharp
public class TodoCompletedConsumer : BackgroundService
{
    private const string ExchangeName = "todo.completed";
    private const string QueueName    = "housing-api.todo-completed";
    // ...
}
```

**Guard conditions before updating — all acked without failure:**

```csharp
// 1. Null deserialization — discard (nack, requeue: false)
if (message is null) { channel.BasicNack(..., requeue: false); return; }

// 2. Missing HousingApplicationId — log and ack
if (string.IsNullOrWhiteSpace(message.HousingApplicationId)) { channel.BasicAck(...); return; }

// 3. Application not found in Mongo — log and ack
var application = _applicationsService.GetAsync(message.HousingApplicationId).GetAwaiter().GetResult();
if (application is null) { channel.BasicAck(...); return; }

// 4. Happy path
_applicationsService.IncrementCompletedReviewAsync(...).GetAwaiter().GetResult();
channel.BasicAck(...);
```

**Why `.GetAwaiter().GetResult()` instead of `await`?**
`EventingBasicConsumer.Received` is a synchronous event callback — it cannot be `async`. The options are:
1. Use `.GetAwaiter().GetResult()` (sync-over-async — acceptable here, one message at a time with `prefetchCount: 1`)
2. Switch to `AsyncEventingBasicConsumer` which requires `factory.DispatchConsumersAsync = true`

Option 1 is simpler and sufficient at this stage. Option 2 is the upgrade path if the consumer needs to handle async I/O more efficiently.

**Why `HousingApplicationsService` is injected directly (not via `IServiceScopeFactory`)?**
`HousingApplicationsService` is registered as a **singleton** (unlike `TodoContext` which is scoped). A singleton is safe to inject into another singleton. No scope factory needed.

## Caveats / what comes next

| Caveat | Impact | Next step |
|---|---|---|
| `.GetAwaiter().GetResult()` in consumer callback | Sync-over-async blocks the thread pool thread; fine with `prefetchCount: 1` but limits throughput | Switch to `AsyncEventingBasicConsumer` with `factory.DispatchConsumersAsync = true` |
| No retry/backoff on consumer startup failure | If RabbitMQ is not ready when `ExecuteAsync` fires, the consumer throws and dies silently | Wrap `RunConsumerLoop` in a retry loop with exponential backoff |
| Nack with `requeue: true` on unexpected exception | A poison message (e.g. DB down) is requeued immediately, causing a tight retry loop | Add dead-letter exchange + DLQ, or add a retry count header check |
| Publisher connects lazily (first PUT that triggers a publish) | First completion fails with 500 if RabbitMQ is unreachable at that moment | Move connection to `IHostedService` with eager connect + reconnect |
| `CompletedReviewCount` not idempotent | If a `TodoCompleted` message is delivered twice (e.g. nack + requeue), the count is incremented twice | Store processed `TodoId` values and skip duplicates, or use `$max` on the timestamp |
| BookStore events not yet wired | `BookCreated` → TodoApi and `TodoCompleted` → BookStoreApi are the remaining roadmap items | Follow the same pattern as this slice |
