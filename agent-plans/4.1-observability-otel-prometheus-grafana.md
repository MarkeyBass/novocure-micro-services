# 4.1 — Observability: OpenTelemetry, Prometheus, Grafana (Traces + Metrics + Logs)

This slice adds full observability to all three .NET services using the OpenTelemetry SDK.
All three signals — **traces, metrics, and logs** — are pushed via OTLP to a single
`grafana/otel-lgtm` container that bundles Grafana, Tempo, Loki, and Prometheus.
RabbitMQ queue metrics are collected via the Prometheus plugin and a dedicated scrape config.

Context: Novocure builds IoT/lab devices (TTFields therapy). The dashboards chosen here
reflect what a lab monitoring platform operator watches — event ingestion rates, pipeline
reliability, service availability, and resource health under continuous telemetry load.

---

## The three signals

| Signal | What it shows | Backend in lgtm | .NET SDK component |
|---|---|---|---|
| **Traces** | Distributed request flow across services, latency per hop, MassTransit spans | Tempo | `AddAspNetCoreInstrumentation` + `AddHttpClientInstrumentation` + `AddSource("MassTransit")` |
| **Metrics** | Request rate, error rate, P95 latency, GC pressure, thread pool depth | Prometheus | `AddAspNetCoreInstrumentation` + `AddRuntimeInstrumentation` + `AddMeter("MassTransit")` |
| **Logs** | Structured log lines with injected `TraceId`/`SpanId` for correlation | Loki | `WithLogging` → OTLP |

**The key feature**: click an error spike in a metrics dashboard → jump to a trace → jump
to the exact log lines for that request. This is the observability story worth demonstrating.

---

## Infrastructure stack

```
.NET services (BookStoreApi, TodoApi, HousingApi)
  │
  │  OTLP gRPC (port 4317)
  ▼
grafana/otel-lgtm ──────────────────────────────────────────────
  │  Tempo    (traces)      → Grafana data source: Tempo
  │  Loki     (logs)        → Grafana data source: Loki
  │  Prometheus (metrics)   → Grafana data source: Prometheus
  └────────────────────────────────────────────────────────────
  Grafana UI: http://localhost:3000

RabbitMQ ──(port 15692: /metrics)──► Prometheus scrape job
  (enabled via rabbitmq_prometheus plugin)
```

The `grafana/otel-lgtm` image is a **learning/dev** image. In production these would
be separate services. The pattern of what gets instrumented and how is identical.

## Full execution flow

Read this before writing any code.

### Step 1 — Add `grafana/otel-lgtm` to docker-compose

Single container. Three ports exposed:
- `3000` → Grafana UI (`admin` / `admin`)
- `4317` → OTLP gRPC (services push telemetry here)
- `4318` → OTLP HTTP (alternative, not used by default)

```yaml
otel-lgtm:
  image: grafana/otel-lgtm:latest
  container_name: otel-lgtm
  restart: unless-stopped
  ports:
    - "3000:3000"
    - "4317:4317"
    - "4318:4318"
  volumes:
    - grafana_data:/var/lib/grafana
    - ./otel-collector-config.yaml:/etc/otelcol-contrib/config.yaml
  networks:
    - app-network
```

Add `grafana_data:` to the `volumes:` section.

### Step 2 — Enable RabbitMQ Prometheus plugin (Option A)

Override the `rabbitmq` service command to enable the plugin at startup.
Expose port 15692 (Prometheus scrape endpoint):

```yaml
rabbitmq:
  image: rabbitmq:4-management
  command: >
    sh -c "rabbitmq-plugins enable rabbitmq_prometheus && rabbitmq-server"
  ports:
    - "5672:5672"
    - "15672:15672"
    - "15692:15692"   # Prometheus /metrics endpoint
  ...
```

After startup, `http://localhost:15692/metrics` returns RabbitMQ queue depth, message rates,
consumer counts, and connection stats in Prometheus exposition format.

### Step 3 — OTel Collector config for RabbitMQ scraping

Create `med-tech-local-infra/otel-collector-config.yaml`.

The `grafana/otel-lgtm` image runs `otelcol-contrib` internally. Mounting a custom config
overrides the default. The config must preserve the OTLP receivers (so .NET services can
still push) and add a Prometheus receiver to scrape RabbitMQ.

```yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318
  prometheus:
    config:
      scrape_configs:
        - job_name: rabbitmq
          scrape_interval: 15s
          static_configs:
            - targets: ['rabbitmq:15692']

processors:
  batch:

exporters:
  otlp/tempo:
    endpoint: localhost:4319   # internal Tempo inside lgtm
    tls:
      insecure: true
  prometheusremotewrite:
    endpoint: http://localhost:9090/api/v1/write  # internal Prometheus inside lgtm
  loki:
    endpoint: http://localhost:3100/loki/api/v1/push  # internal Loki inside lgtm

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlp/tempo]
    metrics:
      receivers: [otlp, prometheus]
      processors: [batch]
      exporters: [prometheusremotewrite]
    logs:
      receivers: [otlp]
      processors: [batch]
      exporters: [loki]
```

> **Note:** The internal port numbers for Tempo, Prometheus, and Loki inside `grafana/otel-lgtm`
> must be verified against the image's default config at runtime
> (`docker exec otel-lgtm cat /etc/otelcol-contrib/config.yaml`).
> If the internal endpoints differ, update the exporters block accordingly.
> This is the only step that may need a quick runtime adjustment.

### Step 4 — Add OTel env vars to all three services in docker-compose

Each service gets three new env vars:

```yaml
OTEL_EXPORTER_OTLP_ENDPOINT: http://otel-lgtm:4317
OTEL_SERVICE_NAME: todo-api          # or bookstore-api / housing-api
OTEL_RESOURCE_ATTRIBUTES: deployment.environment=development
```

Also add `depends_on: otel-lgtm` for each service.

### Step 5 — Add NuGet packages to all three services

Same package set for each `.csproj`:

```xml
<PackageReference Include="OpenTelemetry.Extensions.Hosting" Version="1.11.*" />
<PackageReference Include="OpenTelemetry.Instrumentation.AspNetCore" Version="1.11.*" />
<PackageReference Include="OpenTelemetry.Instrumentation.Http" Version="1.11.*" />
<PackageReference Include="OpenTelemetry.Exporter.OpenTelemetryProtocol" Version="1.11.*" />
<PackageReference Include="OpenTelemetry.Instrumentation.Runtime" Version="1.11.*" />
```

TodoApi only (EF Core traces — in beta, skip if unstable):
```xml
<PackageReference Include="OpenTelemetry.Instrumentation.EntityFrameworkCore" Version="1.0.0-beta.*" />
```

### Step 6 — Wire OTel in Program.cs (identical block in all three services)

Add before `builder.Build()`:

```csharp
builder.Services.AddOpenTelemetry()
    .ConfigureResource(r => r
        .AddService(
            serviceName: builder.Configuration["OTEL_SERVICE_NAME"]
                         ?? builder.Environment.ApplicationName,
            serviceVersion: "1.0.0"))
    .WithTracing(t => t
        .AddAspNetCoreInstrumentation(o => o.RecordException = true)
        .AddHttpClientInstrumentation()
        .AddSource("MassTransit")          // picks up MassTransit publish/consume spans
        .AddOtlpExporter())
    .WithMetrics(m => m
        .AddAspNetCoreInstrumentation()
        .AddHttpClientInstrumentation()
        .AddRuntimeInstrumentation()
        .AddMeter("MassTransit")           // MassTransit message rate metrics
        .AddOtlpExporter())
    .WithLogging(l => l
        .IncludeFormattedMessage(true)
        .IncludeScopes(true)
        .AddOtlpExporter());
```

`AddSource("MassTransit")` + `AddMeter("MassTransit")` are only meaningful in services
that use MassTransit (TodoApi for the BookCreated consumer, BookStoreApi for the publisher).
Include them in all three anyway — they are no-ops in services that don't use MassTransit.

**For EF Core in TodoApi**, add `.AddEntityFrameworkCoreInstrumentation()` inside `WithTracing`.
This produces spans for every SQL query — useful for spotting N+1 patterns.

**For MongoDB in BookStoreApi and HousingApi:** MongoDB Driver 3.x has partial OTel support.
No extra config needed; spans appear automatically if the driver detects an active tracer.

### Step 7 — Log correlation (automatic)

Once `WithLogging` is registered, the OTel SDK injects `TraceId` and `SpanId` into every
log record that is emitted during an active span. No code changes are needed in the
existing `_logger.LogInformation(...)` calls — they are automatically enriched.

In Grafana → Explore → Loki, log lines look like:
```
TraceId=4bf92f3577b34da6a3ce929d0e0e4736 SpanId=00f067aa0ba902b7
INFO: Incremented CompletedReviewCount for HousingApplicationId=...
```

Clicking "Open in Tempo" from a Loki log line navigates directly to the full distributed trace.

### Step 8 — Import Grafana dashboards

Three dashboards to import. Open Grafana at http://localhost:3000 → Dashboards → Import.

| Dashboard | Source | What it shows | IoT relevance |
|---|---|---|---|
| **ASP.NET Core** | Grafana ID `19924` | Request rate, error rate, P95/P99 latency per endpoint | API throughput + SLA compliance |
| **.NET Runtime** | Grafana ID `14058` | GC gen0/gen1/gen2 collections, heap size, thread pool queue | Memory pressure under high telemetry load |
| **RabbitMQ Overview** | Grafana ID `10991` | Queue depth, publish/ack/deliver rates, consumer count | IoT event backlog and pipeline health |

For the RabbitMQ dashboard: select the Prometheus data source and set the `instance` variable
to `rabbitmq:15692`.

If the Grafana ID dashboards use metric names that don't match the OTel SDK output
(common mismatch: `http_server_duration` vs `http.server.request.duration`), use the
community version of the dashboard or create a simple custom panel. Document the working
dashboard IDs in the doc once verified.

## What you see in Grafana after this slice

### Traces (Tempo)

A single `POST /api/applications` request produces a trace like:

```
housing-api  POST /api/applications              45ms
  └── mongodb.insert HousingApplications          8ms
  └── rabbitmq.publish housing.application.created  3ms

todo-api  (consumer) HousingApplicationCreated   22ms
  └── sql.insert TodoItems                        14ms
```

Click any span to see tags: `http.status_code`, `db.statement`, `messaging.destination`.

### Metrics (Prometheus → Grafana dashboards)

**ASP.NET Core dashboard** shows per-endpoint:
- Requests/sec (simulates device telemetry ingestion rate)
- Error rate % (pipeline reliability — critical for medical devices)
- P95 latency (real-time constraint compliance)

**.NET Runtime dashboard** shows:
- GC gen0/1/2 collection frequency (memory pressure under continuous event processing)
- Heap size trend (memory leak detection for long-running services)
- Thread pool queue length (consumer backlog indicator)

**RabbitMQ dashboard** shows:
- Queue depth per queue (is the event backlog growing under load?)
- Publish/ack rates (event throughput)
- Consumer count (are all consumers connected?)

### Logs (Loki)

Every existing `_logger.LogInformation(...)` call gains:
```
traceID=4bf92f3577b34da6a3ce929d0e0e4736 spanID=00f067aa0ba902b7
level=info service=housing-api
msg="Incremented CompletedReviewCount for HousingApplicationId=6601f... (TodoId=3)"
```

Filter in Loki: `{service_name="housing-api"} |= "error"` to find all errors across
the housing service — no log file tailing needed.

## Caveats / what comes next

| Caveat | Impact | Next step |
|---|---|---|
| `grafana/otel-lgtm` is a dev-only image | All backends in one container = single point of failure; not suitable for production | Replace with separate Tempo, Loki, Prometheus containers for a prod-like setup |
| OTel collector internal endpoints may differ from config above | `otel-collector-config.yaml` exporters section needs to match actual lgtm internals | Read default config from running container (`docker exec otel-lgtm cat ...`) and adjust |
| Raw `RabbitMQ.Client` consumers produce no message-level spans | Only ASP.NET Core request spans are traced for the raw flows (slices 3.1–3.4) | Add manual `ActivitySource` instrumentation to `RunConsumerLoop` in each raw consumer |
| EF Core instrumentation is in beta | May not install cleanly on .NET 9 with EF Core 9 | Skip and add when it reaches stable release |
| Metric name format mismatch | OTel SDK uses dotted names; older Grafana dashboards expect underscored names | Find OTel-compatible dashboard IDs or create simple custom panels |
| No alerting configured | Grafana can fire alerts (Slack, email, PagerDuty) when error rate > threshold or queue depth > N | Add alert rules after dashboards are verified — high value for IoT/lab context |
| Traces don't cross the raw RabbitMQ boundary | Trace context is not propagated through raw `RabbitMQ.Client` messages — each side starts a new root span | Inject/extract `traceparent` header manually in publish/consume, or migrate to MassTransit which does this automatically |
