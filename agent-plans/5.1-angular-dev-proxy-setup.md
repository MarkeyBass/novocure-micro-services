# 5.1 — Angular Dev Proxy for Local vs Docker Compose

This note documents the small Angular setup change that removes hardcoded backend URLs from the client and makes it easy to switch between host-based API development and `docker compose` development.

---

## Goal

Keep Angular running on the host with `ng serve`, but make backend routing configurable without editing service files each time.

---

## What changed

| File | Change |
|---|---|
| [`client/src/app/housing.service.ts`](../client/src/app/housing.service.ts) | Replaced hardcoded HousingApi URLs with relative `/api/...` paths |
| [`client/src/app/todo.service.ts`](../client/src/app/todo.service.ts) | Replaced hardcoded TodoApi URL with relative `/api/TodoItems` |
| [`client/proxy.local.json`](../client/proxy.local.json) | Added Angular proxy config for host-run APIs |
| [`client/proxy.compose.json`](../client/proxy.compose.json) | Added Angular proxy config for compose-exposed APIs |
| [`client/package.json`](../client/package.json) | Added `start:local` and `start:compose` scripts; made `start` default to local proxy |

---

## Why this is the better dev workflow

Before this change, the Angular app directly called fixed URLs like:

- `https://localhost:7152`
- `https://localhost:7236`

That caused 2 recurring problems:

1. The client had to be edited when switching between local APIs and compose APIs.
2. The browser could fail with SSL/protocol issues when the frontend expected HTTPS but the container only exposed HTTP.

Using relative paths solves that:

- Angular code always calls `/api/...`
- the Angular dev server proxy decides where those requests go

So environment switching becomes a startup concern, not an application-code concern.

---

## Proxy targets

### Local host development

Used when the APIs are started directly from their projects (for example `dotnet watch`), matching the **`http` profile** URLs in each API’s `launchSettings.json`.

- TodoApi -> `http://localhost:5188`
- HousingApi -> `http://localhost:5125`

Why these are HTTP URLs:

- `https` against a port that only serves plain HTTP (common with Docker-mapped `7152`) produces proxy errors like `EPROTO` / “packet length too long”.
- `ECONNREFUSED` on the HTTPS ports usually means those processes are not running.

Configured in:

- [`client/proxy.local.json`](../client/proxy.local.json)

**If your APIs are actually coming from Docker** (still on the same machine), use `npm run start:compose` instead. Compose maps **HousingApi to `7152`**, not `5125`, so `start:local` and `start:compose` are not interchangeable unless both backends match the chosen ports.

### Docker Compose development

Used when the APIs run inside containers and are exposed through the compose port mappings.

- TodoApi -> `http://localhost:5188`
- HousingApi -> `http://localhost:7152`

Configured in:

- [`client/proxy.compose.json`](../client/proxy.compose.json)

---

## Commands

From `client/`:

### Start against locally run APIs

```bash
npm run start:local
```

### Start against docker-compose APIs

```bash
npm run start:compose
```

### Default start command

```bash
npm start
```

`npm start` currently defaults to the local proxy setup.

---

## Result

The Angular app no longer needs code changes when switching environments.

Only the startup command changes:

- local APIs -> `npm run start:local`
- compose APIs -> `npm run start:compose`

This keeps frontend development simple while avoiding the extra friction of running Angular itself in Docker for day-to-day work.
